Phase 0 — Game design (before touching any code)
This is what people skip the most and what hurts the most to skip. Define the core loop: what resource is generated, what the manual click does, and how it gets automated. Decide the theme (farm, factory, magic, civilization…) because it shapes all naming and art. Sketch the generators (typically 8–12 tiers), the upgrade tree, and the prestige system (the reset that grants meta-currency to accelerate the next run). The most important thing here is the progression curve: base cost of each generator, scaling factor (classically 1.07–1.15), and how many minutes/hours the player should take to reach each milestone. Skip this and you'll do massive rebalancing mid-development. Deliverable: a 2–4 page design doc and a spreadsheet with the simulated economy.

Phase 1 — Stack decisions
For an idle game with a backend, the natural split is frontend running the game loop and backend persisting state, serving accounts and leaderboards. For frontend, React + TypeScript is the most conservative choice; Svelte or Solid perform better in games with many counters updating. For huge numbers (idles routinely hit 1e300+), break_eternity.js or break_infinity.js are the standard — JS's native Number blows up around 1e308. Lightweight backend: FastAPI (Python), Fastify (Node), or Go if you want to learn it. Persistence: PostgreSQL for users and saves, Redis optional for sessions and rate-limiting. Reverse proxy: Caddy (automatic HTTPS) or Traefik (better for Docker).

Phase 2 — Scaffolding and Docker from day 1
Set up the repo as a monorepo (/frontend, /backend, /infra) or split — doesn't matter as long as it's consistent. Create a multi-stage Dockerfile for the frontend (build with Node, serve static files with nginx) and one for the backend (slim image with only what's needed at runtime). The dev docker-compose.yml brings up frontend, backend, Postgres and Redis with hot-reload via mounted volumes. Add a separate compose.prod.yml without volumes, with no host-exposed ports (except the reverse proxy), and pre-built images. Add .dockerignore from the start or your images will balloon to 2 GB and you won't know why. Deliverable: docker compose up works on a fresh clone.

Phase 3 — Playable core loop
Implement the resource counter, the click button, and a fixed tick (typically 20 Hz, i.e. every 50 ms) that updates production. Important: separate the logical state from the render state — update state at 20 Hz but re-render at 60 Hz with interpolation, or your UI will flicker. Implement save serialization/deserialization from the start and persist to localStorage every 10–30 s. Version your save format from v1: you will have to migrate saves when you change the structure, guaranteed. Deliverable: someone can click, watch the number rise, refresh the page, and keep their progress.

Phase 4 — Generators, upgrades and number formatting
Add the generators with their exponential cost formula and production. Implement buy x1 / x10 / x100 / max from the start because without "max" the game is unplayable past one hour. Add a number formatter that goes from normal notation to short suffixes (K, M, B, T) to scientific notation to named numbers (Quadrillion, Quintillion…) depending on preference. Implement the upgrade tree with its multiplicative effects. This is when you'll discover your Phase 0 balancing was off — iterate on the spreadsheet, not by feel.

Phase 5 — Offline progress and prestige
Compute offline production on save load: delta_t = now − last_save, apply accumulated production, show a "you earned X while away" screen. Mind the cap (typically 8–24 h max offline) so you don't break the economy. The prestige system is the heart of replay value: the player resets, keeps a meta-currency proportional to their progress (typically √progress), and spends it on permanent upgrades that speed up the next run. Tune the meta-currency formula so the first prestige is satisfying (~2–4 hours of play) and each subsequent one comes faster.

Phase 6 — Polish and "juice"
Number-up animations, particles on purchase, light screen shake, click sounds with varied pitch, loopable ambient music. "Feel" is half the retention in idles. Unlockable achievements, stats (total clicks, time played, prestiges). No tutorial, but there should be contextual tooltips — new players get lost without them.

Phase 7 — Backend, accounts and cloud saves
So far it's all been client. Now: register/login (email+password, or magic link, or OAuth), endpoints to upload and download the encrypted save, sync between devices. Key decision: does the server validate the save or just store it? If you're going to have leaderboards, it has to validate: re-simulate progress from the last checkpoint using timestamps and check the result is plausible. If it's only a personal save, you can store it opaquely. Rate-limit every endpoint, ideally with Redis. Hash passwords with argon2 or bcrypt — never SHA.

Phase 8 — Anti-cheat and leaderboards (if applicable)
Everything client-side is tamperable; assume anyone can edit their save. Strategies: HMAC-sign saves on the server, validate that deltas are reasonable, detect impossible jumps given elapsed time. For public leaderboards, use cheat-resistant metrics: time-to-first-prestige (measured server-side), not accumulated totals. Consider running an "honor system" unverified leaderboard alongside a stricter "verified" one.

Phase 9 — Deployment
A modest VPS (Hetzner CX22, DigitalOcean, OVH) is enough to start — an idle game isn't CPU-intensive. Install Docker and Docker Compose, copy your compose.prod.yml, put Caddy or Traefik in front with automatic HTTPS via Let's Encrypt. Automated Postgres backups to an S3-compatible bucket (Backblaze B2 or Cloudflare R2 are cheap). CI/CD with GitHub Actions: build images, push to a registry (GHCR is free), webhook the server to docker compose pull && up -d. Centralized logs with Loki+Grafana, or something simpler like Dozzle if it's just you.

Phase 10 — Launch and iteration
Soft launch on r/incremental_games (their community is the target audience and gives brutal but useful feedback), itch.io, and possibly Newgrounds. Minimal privacy-respecting analytics: self-hosted PostHog or Plausible — you don't need Google Analytics for this. Metrics that matter: D1, D7, D30 retention, average time to first prestige, which generator players quit at (tells you where the balancing is broken). From there, an update cycle every 2–4 weeks with new content (more tiers, a new prestige layer, time-limited events).
